# Operational Semantics

The semantics layer specifies the labeled transition system over which every
safety claim is stated. Deposit lifecycle steps are typed transitions whose
preconditions are encoded as constructor arguments, so structurally illegal
moves are type errors rather than runtime failures.

This file describes the operational layer in isolation. The forcing chain that
turns these primitives into the eight-pressure convergence theorem is in
[PROOF-STRUCTURE.md](../PROOF-STRUCTURE.md).

---

## Deposit lifecycle and typed states

A deposit's status is one of `Candidate`, `Deposited`, `Quarantined`,
`Revoked`, or `Forgotten`. The transitions between these states are not
suggestions — every `Step` constructor that changes a deposit's status takes
the previous status as an explicit hypothesis (`isDeposited`, `isQuarantined`,
`isCandidate`).

Once a deposit is `Deposited`, the structured lifecycle offers two exits:
`Challenge` (→ `Quarantined`) or `Forget` (→ `Forgotten`). There is no
structured constructor that exits a `Deposited` deposit back to `Candidate`.
`Update` can overwrite any non-`Forgotten` slot to any status, but it opts
the trace into the revision predicate (`Action.isRevision = true`). The
absorbing-state behaviour the forcing chain uses to derive the revocation
primitive is stated over revision-free traces, so `Update` does not undercut it.

## Action surface and the LTS

The bank-side action surface is the inductive `Action`, with one constructor
per agent-initiated operation plus a single environment action `Tick`:

```
Submit · Register · Withdraw · Challenge · Repair · Revoke ·
Promote · Forget · Update · Tick
```

Every agent-initiated constructor records `(a : Agent) (B : Bubble)` so that
traces are fully attributed. `Tick` carries only a clock-monotonicity witness
`s.clock ≤ t'`. There are no ACL tables, bank-authority lists, or trust-bridge
registries inside `SystemState`: trust is per-deposit (`d.h.acl`), and
authorization is an agent-level concern.

The LTS itself is the inductive `Step : SystemState → Action → SystemState →
Prop`. Ten constructors total — `submit` produces a `Candidate`, `register`
produces a `Deposited`, and the remaining eight are one-to-one with their
`Action` variant. Multi-step reachability is the inductive `Trace s s'`
generated by `nil` and `cons`.

## Precondition table

Preconditions are pure structural ledger reads. The full table:

| Action | Precondition |
|---|---|
| `Submit` | none — enters as `Candidate` |
| `Register` | none — enters as `Deposited` |
| `Withdraw` | `isDeposited` |
| `Challenge` | `isDeposited` |
| `Promote` | `isCandidate` |
| `Repair` | `isQuarantined` |
| `Revoke` | `isQuarantined` |
| `Forget` | slot exists, `d_old.status ≠ Forgotten` |
| `Update` | slot exists, `d_old.status ≠ Forgotten` |
| `Tick` | `s.clock ≤ t'` |

Because the type system enforces the table, safety invariants are *maintained
by construction*: once step-level preservation is proved for an invariant
`Inv : SystemState → Prop`, trace-level preservation follows generically from
`generic_invariant_preservation` — no per-trace case analysis.

## Gate theorems

The withdrawal gate is the canonical example. `isDeposited` is a pure
status predicate (`∃ d, ledger.get? d_idx = some d ∧ d.status = .Deposited`)
— it checks the ledger slot, nothing else. The withdrawal-gate theorem in
`Theorems/Withdrawal.lean` is the structural projection: if `Step.Withdraw`
fires, then ACL permission, current τ, and `Deposited` status all held in
the source state. The ACL/τ conditions are what the gate theorem documents;
they are not checked by `isDeposited` itself.

The same pattern produces `Repair → Candidate` (revalidation), `Promote
requires Candidate`, and the `Forget` / `Update` slot-existence gates. These
sit in `Theorems/Withdrawal.lean` and are universally quantified over every
`SystemState`/`Step` instance — no model parameter varies, so the LTS gates
are part of the always-on theorem surface (cf. the registry in
[reference/THEOREMS.md](../reference/THEOREMS.md)).

## Repair, revision, and the `Update` constructor

`Repair` does not mark a quarantined deposit as fixed. It returns it to
`Candidate`, so the lifecycle has to re-cross the `Promote` gate before the
deposit is `Deposited` again. Revalidation is forced.

`Update` is the wholesale-replacement constructor: the slot's deposit becomes
`d_new` in its entirety. The action is classified as a revision action by
`Action.isRevision`, so any trace containing `Update` has
`Trace.hasRevision = true`. Revision safety theorems
(`Semantics/RevisionSafety.lean`) reason about exactly this constructor.

## What the semantics layer does not claim

This file specifies *transition structure and precondition encoding*, not the
choice of agent strategies, not the eight-pressure convergence chain, and not
the existence of a witness model. The convergence chain is in
[PROOF-STRUCTURE.md](../PROOF-STRUCTURE.md); the world-side semantic interface
that obligation theorems quantify over is in [WORLD.md](WORLD.md).

`AgentLTS` (in `Agent/Resilience.lean`) is a deliberately coarser
over-approximation of `Step`; it exists so that containment proofs are easier
to state, and results transfer back to `StepSemantics` via the
`StepSemantics ⊆ AgentLTS` simulation. Both are operational; neither is a
runtime implementation.
